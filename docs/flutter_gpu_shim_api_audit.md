# flutter_gpu_shim API surface audit

Scoping how much of `package:flutter_gpu` the shim has to implement to make
flutter_scene compile and run on web (WebGL2).

Generated by grepping `packages/flutter_scene/lib/` + `examples/flutter_app/lib/`
for `gpu.X` patterns. Master at commit time of writing.

## Headline numbers

- **15 RenderPass methods** used (no stencil, no scissor, no polygon-mode)
- **8 GpuContext methods/getters** used
- **2 CommandBuffer methods** used
- **3 buffer/texture upload methods** used (`emplace`, `overwrite`)
- **2 ShaderLibrary entry points**, **1 Shader method** used
- **~24 enum values** across PrimitiveType / PixelFormat / StorageMode / etc.
- **No compute shaders.** No texture arrays. No cubemaps. No 3D textures.

## RenderPass

Used: `bindPipeline`, `bindVertexBuffer`, `bindIndexBuffer`, `bindUniform`,
`bindTexture`, `clearBindings`, `draw`, `setColorBlendEnable`,
`setColorBlendEquation`, `setCullMode`, `setDepthCompareOperation`,
`setDepthWriteEnable`, `setPrimitiveType`, `setViewport`, `setWindingOrder`.

Not used: `setStencilReference`, `setStencilConfig`, `setPolygonMode`,
`setScissor`. Stencil and scissor can be deferred until/unless flutter_scene
or a downstream consumer adds them.

## Buffer / texture data upload

- `HostBuffer.emplace(ByteData)` — transient (per-frame) uniform data
- `DeviceBuffer.overwrite(ByteData, {destinationOffsetInBytes})` — static vertex/index data
- `Texture.overwrite(ByteData)` — CPU-side texture pixel data

## Shaders

- `gpu.ShaderLibrary.fromAsset(path)` — load the `.shaderbundle` flatbuffer
- `gpu.Shader.getUniformSlot(name)` — resolve slot by reflected name
- `gpu.BufferView(buffer, offsetInBytes, lengthInBytes)` — for indirecting into a buffer when binding

## Capabilities flutter_scene needs

| Need | WebGL2 support | Risk |
|---|---|---|
| HDR float color targets (`r16g16b16a16Float`) | `EXT_color_buffer_float` — widely available | Low |
| Linear filtering on float textures | `OES_texture_float_linear` — modern devices OK, not universal | Medium |
| 32-bit float shadow target (`r32g32b32a32Float`) | `EXT_color_buffer_float` covers this | Low |
| MSAA + resolve to sampleable texture | `glRenderbufferStorageMultisample` + `glBlitFramebuffer` to a non-MSAA color texture | Low — standard WebGL2 |
| 256-byte UBO alignment | WebGL2 reports per-driver; usually 256, sometimes higher. Need a runtime check | Low |
| Indexed drawing (int16 + int32) | Built-in | Low |
| Vertex attribute layout (48B / 80B) | VAOs cover it | Low |
| Per-draw blend / depth / cull state | Built-in via GL state | Low |
| Per-draw viewport | `glViewport` | Low |
| Compute shaders | **N/A — not used.** Env prefilter is fragment-shader-driven. | None |
| Texture arrays / cubemaps | **N/A — not used.** IBL atlas + cascade map are 2D. | None |

## Shape of work

Conceptually the shim has to provide:

1. `GpuContext` with 7 factory methods + 2 getters.
2. `CommandBuffer` (queue of render passes, submit replays them on the GL context).
3. `RenderPass` translating bind/state/draw calls into GL state mutations + the actual draw call.
4. `Shader` / `ShaderLibrary` parsing the Impeller `.shaderbundle` flatbuffer and pulling out the OpenGLES variant.
5. `DeviceBuffer` (a VBO/UBO/IBO with `overwrite`), `HostBuffer` (a ring of UBO segments handed out by `emplace`).
6. `Texture` (a 2D texture or color-attachable renderbuffer) with `overwrite`.
7. The small RenderTarget / ColorAttachment / DepthStencilAttachment record types — already trivial in the existing shim repo.

The hardest part is (3) — encoding the bind-by-name uniform/texture metadata
into GL uniform locations and UBO bind points, using the reflection data in
the shader bundle. The existing flutter_gpu_shim repo already wires the
reflection metadata into `_UniformBinding` / `_TextureBinding` structures
parsed from the flatbuffer; that work transfers over.

## What's NOT in scope (good news)

- No compute, texture arrays, cubemaps, or 3D textures.
- No stencil (no shader-driven stencil; no `setStencil*`).
- No scissor (no `setScissor`).
- No wireframe / polygon mode.
- Only one `CompareFunction` variant (lessEqual) and one `WindingOrder`
  variant (CCW) actually appear in flutter_scene, so even the depth/winding
  surface could ship with stubs for the rest initially.

## Open questions for the shader experiment

- Per-shader reflection metadata: are `set` / `binding` / `extRes0` numbers
  populated for the OpenGLES backend, or only for Vulkan/Metal?
